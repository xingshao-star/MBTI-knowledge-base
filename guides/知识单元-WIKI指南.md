# 知识单元 → Wiki 编译指南

> 本指南是从知识单元编译生成 Wiki 页面的唯一执行指南。
> 仅在编译 Wiki 时使用。查询相关规则见 `guides/LLM查阅知识库指南.md`。

---

## 〇、编译前准备：确定来源范围

每次编译以一个来源文件夹为单位执行：

1. **指定来源文件夹**：确认本次编译的来源文件夹路径（如 `units/ENFJ-Protagonist主人公/`）
2. **读取来源 index.md**：阅读该文件夹下的 `index.md`，获取本次需要编译的知识单元完整清单
3. **按清单逐一编译**：依照下方规则，将每个知识单元发布或编译到对应的 Wiki 页面

### 并行执行规则

当待编译的页面数量超过 6 个时，须创建多个 Subagent（子代理）并行执行：

- **Subagent 类型**：`general-purpose`，model 固定为 `sonnet`
- **每个 Subagent 最多处理 6 个页面**
- **分配方式**：按知识单元清单顺序均匀分配，每个 Subagent 的 prompt 中须包含：本指南路径、目标来源文件夹路径、分配给该 Subagent 的知识单元列表
- **6 个及以下页面**：无需创建 Subagent，直接执行

## 一、Wiki 分类体系

Wiki 采用大分类结构，每个分类对应一个目录。`wiki/wiki-index.md` 作为语义目录入口。

| 分类 | 目录 | 内容说明 |
|------|------|----------|
| 体系总览 | `wiki/theory/` | MBTI 理论的叙事性综述（800-1500 字），读完即可建立整体认知 |
| 理论背景 | `wiki/theory/` | MBTI 理论历史、方法论等独立理论页面 |
| 人格类型 | `wiki/types/{TYPE}/` | 每个类型一个子目录，含枢纽页（总览+导航）和多个详情页（1:1 对应知识单元） |
| 四大维度 | `wiki/dimensions/` | E-I、S-N、T-F、J-P 及 A-T 身份维度深度解析 |
| 认知功能 | `wiki/functions/` | Se/Si/Ne/Ni/Te/Ti/Fe/Fi 八大认知功能 |
| 类型组 | `wiki/type-groups/` | NT 分析家、NF 外交家、SJ 守卫者、SP 探险家 |
| 策略 | `wiki/strategies/` | 四大策略（心智维度×身份维度的交叉产物） |
| 类型关系 | `wiki/relationships/` | 组级别关系（如 ENFJ-NT组-友谊）和个体级别关系（如 ENFJ-INTJ-友谊） |
| 个人成长 | `wiki/growth/` | 跨类型的成长专题（类型专属成长内容在 types/ 下） |
| 生活领域 | `wiki/life-domains/` | 跨类型的生活领域专题（类型专属内容在 types/ 下） |
| 对比分析 | `wiki/comparisons/` | 类型间对比、维度对比等分析页面 |
| 来源摘要 | `wiki/sources/` | 每份原始素材的摘要页 |

## 二、人格类型页面规范

每个类型在 `wiki/types/` 下有独立子目录（如 `wiki/types/ENFJ/`），包含一个**枢纽页**和多个**详情页**。

### 枢纽页（Hub Page）

> 创建枢纽页时，以 `templates/type-hub.md` 为骨架模板。

枢纽页（`wiki/types/{TYPE}/{TYPE}.md`）是该类型的导航中心，不是内容页。职责：

1. **精炼总览**（300-500 字）：从 `units/{TYPE}/概述-谁是{TYPE}.md` 提炼，让读者 30 秒建立基本认知
2. **认知功能栈速查表**
3. **详情页分类导航**：列出所有详情页链接，每条附一句话说明。说明须通俗易懂、全面展示该详情页主要涵盖的内容（而非仅重复标题），目的是让 LLM 检索时快速匹配用户问题
4. **所属分类标注**：标注该类型所属的类型组、策略、相关维度（纯文本，**不使用双链**，因为这些是同级/跨类别页面）
5. **相关页面**：仅链接 `[[wiki-index]]`（父级），不链接指南文件或其他非父级页面

### 详情页（Detail Page）

> 创建详情页时，以 `templates/type-detail.md` 为骨架模板。

详情页（`wiki/types/{TYPE}/{TYPE}-{category简写}-{topic}.md`）与知识单元 1:1 对应，每页聚焦一个主题。

**命名规则**：`{TYPE}-{category简写}-{topic}.md`，在整个 vault 中全局唯一。

示例：
- `ENFJ-概述-谁是ENFJ.md`
- `ENFJ-成长-自尊.md`
- `ENFJ-育儿-婴儿期.md`
- `ENFJ-职业-总论与选择.md`

详情页的内容来自知识单元的轻量发布（见第四节），正文近乎原文保留。

## 三、index.md 语义目录规范

`wiki/wiki-index.md` 是 LLM 查阅的首要入口。采用**语义目录**格式，每个条目包含足够信息让 LLM 不打开页面即可判断相关性。

### 设计原则

LLM 读取 index.md 的目的是：**在不打开任何页面的情况下，判断哪个页面最可能回答当前问题**。

### 条目格式

```markdown
- [[页面名]] — {30-50 字内容摘要，包含核心关键词}
```

摘要必须包含：
- 该页面的核心主题关键词
- 该页面覆盖的具体内容范围
- 区分同类页面的差异点

### 结构

```markdown
# MBTI 知识库 — 语义目录

> LLM 查阅入口。阅读本文件定位目标页面，再进入页面获取详情。
> 每条目含内容摘要与关键词，无需打开页面即可判断相关性。

## 体系总览
- [[MBTI理论-体系总览]] — {摘要}
- [[MBTI理论-历史背景]] — {摘要}

## 人格类型（types/）
> 每个类型有枢纽页（总览+导航）和多个详情页

### {类型组名}
**{TYPE} — {中文名}**
- [[{TYPE}]] — 枢纽页：{摘要}，含 N 个详情页及 M 个关系页

## 四大维度（dimensions/）
- [[{维度页}]] — {摘要}

## 类型组（type-groups/）
...

## 策略（strategies/）
...

## 类型关系（relationships/）
> 详细页面通过类型枢纽页导航，index.md 中只列出入口说明
- [[{TYPE}]] → 枢纽页中"类型间关系"区块，含 N 页

## 来源摘要（sources/）
...
```

当某个分类下页面较多时（超过 30 个），在该目录内创建子索引文件（如 `wiki/types/_index.md`），wiki-index.md 中只列出分类入口链接。

## 四、发布与编译规则

### 两种模式

| 模式 | 触发条件 | 操作强度 |
|------|----------|---------|
| **轻量发布** | 该主题仅一个知识单元（单来源） | 替换 frontmatter + 添加摘要 + 添加相关页面 |
| **综合编译** | 同一主题有 ≥2 个知识单元（多来源） | 完整编译 + 共识机制（见第五节）+ 多视角合并 |

### 轻量发布步骤

**步骤 1**：读取知识单元，确认 `_source` 只有一个来源。

**步骤 2**：转换 frontmatter——将知识单元的溯源型 frontmatter 替换为 Wiki 的读者型 frontmatter：

| 知识单元字段 | Wiki 页面字段 | 转换规则 |
|-------------|-------------|---------|
| `id` | `sources: [id]` | id 值移入 sources 列表 |
| `topic` | `title` | 格式化为 `"{TYPE} — {topic}"` |
| `tags`（去除 `_` 前缀字段） | `tags` | 保留，补充 `mbti` 和 `{category}` |
| （无） | `created` / `updated` | 填入当前日期 |
| （无） | `status` | 固定为 `draft` |

**步骤 2.5**：添加一句话摘要——在 frontmatter `---` 结束行之后、正文第一个标题之前，插入一行摘要：

```markdown
> {30-50 字摘要，概括本页核心内容，供 LLM 快速判断相关性}
```

摘要前后各空一行。撰写要求：
- **通俗易懂**：避免抽象概括，用具体内容说明页面涵盖什么
- **全面展示主要内容**：摘要应覆盖页面的主要知识点和涉及的方面，而非仅提及主题名称
- **便于匹配**：包含用户可能提问时使用的关键词，让 LLM 读一行即可决定是否需要阅读全文

示例对比：
- 不合格：`> ENFJ 的职业发展相关内容`
- 合格：`> ENFJ 的职业选择偏好、适合的工作环境类型、团队中的协作风格、领导力表现与管理岗位适配性`

**步骤 3**：保留正文，**不添加行内双链**：
- 正文中不对 MBTI 术语或其他页面名称添加 `[[双链]]`
- 所有页面间链接集中在 `## 相关页面` 区块中管理
- **不改动实质内容、不删减段落、不重组结构**

**步骤 4**：添加 `## 相关页面` 区块（严格层级链接）：

链接规则：**只允许上下级链接，禁止同级链接和跨类别链接**。所有页面只能链接到其直接父级。

| 页面类型 | 父级链接 | 链接数 |
|----------|---------|--------|
| 枢纽页 | `[[wiki-index]]` | 1 |
| 类型详情页 | `[[{TYPE}]]`（枢纽页） | 1 |
| 关系页 | `[[{TYPE}]]`（枢纽页） | 1 |
| 维度页 | `[[wiki-index]]` | 1 |
| 类型组页 | `[[wiki-index]]` | 1 |
| 策略页 | `[[wiki-index]]` | 1 |
| 理论页 | `[[wiki-index]]` | 1 |
| 来源摘要页 | `[[{TYPE}]]` 或 `[[wiki-index]]` | 1 |

**禁止链接到**：
- **同级页面**：同分类目录下的其他页面（如维度页之间、类型组页之间、策略页之间、详情页之间、关系页之间）
- 不同分类目录的页面（如详情页不链接维度页、类型组页）
- 非直接上下级的页面

**步骤 5**：写入目标路径（按目录结构）。

**步骤 6**：在 `wiki/wiki-index.md` 注册（格式见第三节）。

注册规则：
- **类型详情页和关系页不在 wiki-index.md 中逐一注册**，只注册枢纽页。详情页和关系页的导航由枢纽页承担。
- 新增类型时，在 wiki-index.md 的"人格类型"部分添加枢纽页链接，附注详情页和关系页总数。
- 维度页、类型组页、策略页、理论页、来源摘要页在 wiki-index.md 中逐一注册。

**步骤 7**：在 `wiki/log.md` 追加记录。

| `friendship` | `wiki/types/{TYPE}/{TYPE}-友谊-*.md` | 轻量发布为详情页 |
| `parenting` | `wiki/types/{TYPE}/{TYPE}-育儿-*.md` | 轻量发布为详情页 |

| `career` | `wiki/types/{TYPE}/{TYPE}-职业-*.md` | 轻量发布为详情页 |

### 关系单元映射

关系单元分为两种类型：

| 类型 | 命名格式 | 目标路径 | 发布方式 |
|------|---------|---------|---------|
| 组级别（如 ENFJ-NT组-友谊） | `{TYPE}-{对方组}-{场景}.md` | `wiki/relationships/` | 轻量发布，完整独立主题 |
| 个体级别（如 ENFJ-INTJ-友谊） | `{TYPE-A}-{TYPE-B}-{场景}.md` | `wiki/relationships/` | 综合编译，需双视角合并 |

组级别关系页描述一个类型与某个类型组的整体互动模式，是完整的独立主题，直接发布。
个体级别关系页需要双方类型均收录后再创建。

### 通用理论单元映射

通用理论内容（维度、类型组、策略、理论历史）不再区分"共享单元"，按常规知识单元处理。当多个来源文件夹包含同一主题的通用理论单元时，走综合编译流程。

| 单元类别 | 目标路径 |
|----------|---------|
| 维度 | `wiki/dimensions/` |
| 类型组 | `wiki/type-groups/` |
| 策略 | `wiki/strategies/` |
| 理论历史 | `wiki/theory/` |

### 综合编译原则（多来源场景）

当第二个及后续来源的知识单元编译到已有 Wiki 页面时：

1. **整合而非复制**：Wiki 页面是多个知识单元的提炼与交叉引用产物
2. **交叉引用**：使用 `[[页面名]]` 双链语法在 `## 相关页面` 区块中建立页面间关联，遵循严格层级链接规则（见第四节步骤 4）
3. **多来源综合**：同一主题的多个知识单元应综合为一个 Wiki 段落，遵循共识机制（见第五节）
4. **精炼优先**：Wiki 页面追求信息密度，避免冗余表述
5. **术语一致**：所有专业术语须参照 `docs/MBTI术语表.md` 中的标准译法

### 来源摘要页创建

每份来源完成知识单元拆解并发布到 Wiki 后，必须在 `wiki/sources/` 中创建对应的来源摘要页：

1. 以 `templates/source-summary.md` 为骨架模板
2. `source_units` 字段列出该来源产出的所有知识单元 id（从来源文件夹的 index.md 获取）
3. 分配来源简称（S1、S2、S3……按收录顺序递增），作为共识标记中的引用标识
4. 在 `wiki/wiki-index.md` 的"来源摘要"分类中注册
5. status 设为 `draft`

> 来源摘要页是共识机制中 `S1, S2, S3` 引用的基础，必须在发布阶段同步创建。

## 五、观点共识机制

### 设计目的

当多份来源资料涉及同一知识点时，需要区分哪些是多方共识、哪些是单一来源的补充、哪些存在分歧。通过动态的共识级别标记，让 Wiki 内容的可信度层次清晰可辨。

### 共识级别定义

| 级别 | 标记 | 含义 | 触发条件 |
|------|------|------|----------|
| **共识** | `[共识]` | 多方来源一致认同的观点 | ≥3 个独立来源持相同观点 |
| **多数** | `[多数]` | 多数来源认同，但未达广泛共识 | 2 个独立来源持相同观点 |
| **单源** | `[单源]` | 仅一个来源提出的观点 | 首次收录时的默认级别 |
| **分歧** | `[分歧]` | 不同来源持不同观点 | 来源间观点存在实质性差异 |

### 标记语法

在 Wiki 页面中，对每个关键观点使用以下格式：

```markdown
INFJ 通常是极其尽责的父母，将对人类未来的希望寄托在子女身上。[共识 · S1, S2, S3]

INFJ 在面对子女叛逆期时可能因理想与现实的落差而过度焦虑。[单源 · S2]

> [分歧] 关于 INFJ 父母是否应采用严格管教：
> - **多数观点**（S1, S3）：INFJ 不适合严格管教，应以引导和共情为主
> - **少数观点**（S2）：适度的规则设定对 INFJ 父母和子女都是必要的成长
```

其中 `S1, S2, S3` 为来源摘要页的简写引用（对应 `wiki/sources/` 中的页面）。

### 新来源收录时的比对规则

当编译新来源的知识单元到已有 Wiki 页面时，对每个观点执行以下判断：

```
对于新来源中的每个观点 P：
  │
  ├─ Wiki 中是否已有相同观点？
  │   │
  │   ├─ 是 → 情况 A：观点一致
  │   │   → 将该观点的来源列表追加新来源
  │   │   → 重新评估共识级别（见升级规则）
  │   │   → 如新来源有更精准的表述，可优化措辞
  │   │
  │   └─ 否 → Wiki 中是否有相关但不同的观点？
  │       │
  │       ├─ 否 → 情况 B：全新补充
  │       │   → 新增该观点，标记为 [单源 · 新来源]
  │       │
  │       └─ 是 → 差异程度如何？
  │           │
  │           ├─ 补充性差异（不矛盾，是不同角度） → 情况 C
  │           │   → 新增为独立观点 [单源 · 新来源]
  │           │   → 与已有观点建立关联（同一段落或相邻段落）
  │           │
  │           └─ 实质性分歧（观点矛盾） → 情况 D
  │               → 将已有观点和新观点合并为 [分歧] 格式
  │               → 分别标注各方来源
  │               → 如已有观点原为 [共识] 或 [多数]，保留其级别描述
```

### 共识级别动态调整规则

**升级规则：**

| 当前级别 | 新来源态度 | 操作 |
|----------|-----------|------|
| `[单源]` | 新来源持相同观点 | 升级为 `[多数 · S1, S2]` |
| `[多数]` | 新来源持相同观点 | 升级为 `[共识 · S1, S2, S3]` |
| `[共识]` | 新来源持相同观点 | 保持 `[共识]`，追加来源 |
| `[分歧]` 中的某方 | 新来源支持该方 | 该方观点标注为"多数观点"或"共识观点"（见下方分歧演化规则） |

**分歧演化规则：**

当一个 `[分歧]` 标记的观点中，某一方获得更多来源支持时：

```markdown
# 初始状态（2 个来源各执一词）
> [分歧] 关于 INFJ 是否适合管理岗位：
> - 观点 A（S1）：INFJ 适合，因其共情力和远见
> - 观点 B（S2）：INFJ 不适合，因其回避冲突的倾向

# S3 收录后支持观点 A → 观点 A 升级为多数
> [分歧] 关于 INFJ 是否适合管理岗位：
> - **多数观点**（S1, S3）：INFJ 适合管理岗位，因其共情力和远见能凝聚团队
> - 少数观点（S2）：INFJ 可能因回避冲突的倾向而在管理中遇到挑战

# S4 也支持观点 A → 观点 A 升级为共识
```

## 六、页面状态管理

| 状态 | 含义 | 判定条件 |
|------|------|----------|
| `draft` | 收录编译中创建 | LLM 执行收录流程时系统性创建的页面，内容可能尚未覆盖所有相关知识单元 |
| `active` | 已审核确认 | 人类审阅通过后手动设置。仅人类可触发此状态 |
| `unreviewed` | 查询动态创建 | LLM 在回答用户问题时自动创建的新页面（由查阅指南触发，非本指南范围） |
| `review` | 需重新审核 | 已有 active 页面因新来源收录导致实质性内容变更 |

编译过程中创建的页面一律使用 `draft` 状态。当新来源导致已有 `active` 页面发生实质性变更时，将其改为 `review`。

## 七、新类别创建规则

当知识单元中出现无法归入现有 Wiki 分类的主题时：

1. **识别**：确认该主题确实不属于任何现有分类
2. **评估**：判断是否有足够内容支撑一个独立分类（至少 3 个相关知识单元）
3. **创建**：
   - 在 `wiki/` 下创建新目录
   - 在 `wiki/wiki-index.md` 中注册新分类
   - 新分类下的页面初始 status 为 `draft`
4. **不足以独立成类时**：暂时归入最相近的现有分类，在页面中标注"待归类"

## 九、页面篇幅控制与拆分规则

### 字数上限

Wiki 页面正文（不含 frontmatter 和相关页面区块）不超过 **2500 字**。生成页面时若超过此限制，须评估是否拆分。

### 拆分判断标准

对页面内容进行"一句话总结"测试，若满足以下任一条件则应拆分：

1. **多概念**：一句话总结中包含了多个独立概念或主题
2. **多领域**：内容可细分为多个领域知识，各领域可独立成篇
3. **多问题**：内容可以回答用户多个不同性质的问题（而非同一问题的不同层面）

满足拆分条件时，将页面拆分为多个聚焦单一主题的详情页，并在枢纽页中分别导航。

### 不适合拆分的情况

若内容虽超过 2500 字，但属于同一主题的连贯论述、拆分后各部分无法独立成篇，则不拆分，改为**提炼精简**，将字数控制在 2500 字以内。提炼时保留核心观点和关键论据，删减冗余表述和重复举例。

### 枢纽页导航描述规范

在枢纽页中用一句话描述各详情页时，须**充分体现该页面主要涵盖的内容**，目的是让 LLM 在检索时能快速匹配用户问题与目标页面。避免泛泛的标题式描述，应包含核心关键词和内容范围。

## 十、体系总览页（MBTI理论-体系总览.md）内容规范

### 定位

`wiki/theory/MBTI理论-体系总览.md` 是整个知识库的**叙事性综合入口**。读完此页即可对 MBTI 体系建立完整的基本认知。

### 内容要求

- **篇幅**：800-1500 字
- **风格**：叙事性散文，不是表格罗列；有上下文、有逻辑串联
- **结构**：
  1. 什么是 MBTI（起源、核心理念）
  2. 四大维度 + 身份维度（每个维度 2-3 句叙述）
  3. 16 型人格与四大类型组（分组逻辑）
  4. 四大策略（维度交叉产物）
  5. 认知功能栈（概念介绍）
  6. 类型模型的价值与局限
- **链接**：每个概念首次出现时链接到对应 Wiki 页面
- **演化**：每次收录新来源后评估是否需要更新（见下方触发条件）

### 更新触发条件

| 触发条件 | 操作 |
|----------|------|
| 新来源引入了体系总览页未覆盖的重要概念或分类 | 在体系总览页对应位置补充说明 |
| 新来源对已有描述提供了更精准的表述 | 优化体系总览页中的相关措辞 |
| 新来源涉及新的维度、类型组或应用领域 | 在体系总览页中增加对应段落 |
| 新来源仅补充了已有页面的细节 | 无需更新体系总览页 |

更新体系总览页后，将其 status 改为 `review`，在 `wiki/log.md` 中记录更新内容。
