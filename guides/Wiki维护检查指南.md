# Wiki 维护检查指南

> 本指南是对 Wiki 进行健康检查与维护的唯一执行指南。
> 仅在用户要求维护检查或一轮完整收录编译完成后使用。

---

## 一、检查触发时机

| 时机 | 说明 |
|------|------|
| 用户主动要求 | 用户明确要求进行 Wiki 维护、检查、清理 |
| 收录编译完成后 | 一份或多份来源完成"原始资料→知识单元→Wiki"全流程后 |
| 积累一定量 unreviewed 页面时 | 查询过程中产生的 unreviewed 页面需要集中处理 |

---

## 二、按状态分级检查

### 检查优先级

```
review（最高）→ unreviewed → draft → 结构完整性
```

### 2.1 review 页面（优先级最高）

**含义**：已审核的 active 页面因新来源收录导致实质性变更。

**检查项**：

1. **新增内容准确性**：新来源带来的内容是否准确整合到页面中
2. **共识标记更新**：来源数量与共识级别是否匹配（单源/多数/共识/分歧）
3. **交叉引用完整**：新增内容涉及的双链是否已建立
4. **与原有内容一致性**：新增部分是否与页面已有内容逻辑连贯，无矛盾

**产出**：逐页列出检查结果，修正发现的问题后提示用户重新审核。

### 2.2 unreviewed 页面

**含义**：LLM 在回答用户问题时自动记录的新页面，尚未经人类审核。

**检查项**：

1. **知识单元支撑**：页面内容是否有对应的知识单元作为依据，还是纯粹基于 LLM 自身知识生成
2. **与已有页面关系**：是否与现有页面重复或高度重叠
3. **内容质量**：信息是否准确、结构是否符合模板规范
4. **归属判断**：该页面应保留在当前目录，还是应合并到已有页面

**产出**：对每个 unreviewed 页面给出处理建议：

| 建议 | 条件 | 操作 |
|------|------|------|
| **保留** | 内容有知识单元支撑，主题独立，不与已有页面重复 | 完善内容后提示用户审核 |
| **合并** | 内容与已有页面高度重叠 | 将有价值的部分合并到目标页面，删除原页面 |
| **补充** | 页面主题有价值但内容单薄 | 标注需要补充的方向，提示用户补充来源资料 |
| **删除** | 无知识单元支撑且内容质量低 | 建议用户确认后删除 |

### 2.3 draft 页面

**含义**：收录编译流程中系统性创建的页面，内容可能尚未覆盖所有相关知识单元。

**检查项**：

1. **知识单元覆盖度**：对照 `units/index.md`，该页面是否已涵盖所有相关知识单元的内容，有无遗漏
2. **主题针对性**：页面是否聚焦于自身主题，而非泛泛而谈
3. **差异性**：与同类页面（如同一目录下的其他页面）是否有实质性区分，避免内容雷同
4. **结构规范**：是否符合对应模板的结构要求
5. **共识标记**：多来源内容的共识级别标记是否正确
6. **篇幅与拆分评估**：正文超过 2500 字的页面，须评估是否应拆分。判断标准：用一句话总结内容，若包含多个独立概念/主题、可细分为多个领域知识、或可回答用户多个不同性质的问题，则应拆分为多个详情页。不适合拆分的，提炼精简至 2500 字以内

**产出**：逐页给出完善建议，标注缺失的知识单元引用和需要补充的内容方向。

### 2.4 active 页面

**规则**：默认已审核，无需检查。

**例外**：当其他检查项发现 active 页面存在以下问题时，标记为 `review`：
- 其他页面的交叉引用指向该页面中不存在的段落
- 新收录的知识单元与该页面内容存在矛盾
- wiki-index.md 中该页面的说明与实际内容不符

## 三、内容质量检查

在完成按状态的分级检查后，对所有非 active 页面执行以下质量检查：

### 3.1 知识单元覆盖度

- 读取 `units/index.md`，定位各来源文件夹，再读取各来源 index.md 列出所有已有知识单元
- 对照 Wiki 页面，检查是否有知识单元的内容未被任何 Wiki 页面引用
- 遗漏的知识单元应标注到对应 Wiki 页面的完善建议中

### 3.2 主题针对性与差异性

- 同一目录下的页面（如 `types/` 下的 16 个类型页面）应各有侧重
- 检查是否存在大段相似表述（复制粘贴式内容）
- 共享性内容应提炼到上层页面（如枢纽页、`type-groups/`），类型页面保留该类型的独特表现

### 3.3 共识标记一致性

- `[共识]` 标记的观点是否确实有 ≥3 个来源
- `[多数]` 标记的观点是否确实有 2 个来源
- `[分歧]` 标记中各方来源是否正确列出
- `[分歧·倾向明确]` 的判定是否合理

### 3.4 枢纽页导航描述质量

- 检查枢纽页中对各详情页的一句话描述是否**充分体现该页面主要涵盖的内容**
- 描述应包含核心关键词和内容范围，让 LLM 在检索时能快速匹配用户问题与目标页面
- 避免泛泛的标题式描述（如"关于职业的内容"），应具体说明涵盖哪些方面（如"职业选择偏好、适合的工作环境、团队协作风格与领导力表现"）
- 不合格的描述标注到该页面的完善建议中，并直接修正

### 3.5 术语一致性

- 参照 `docs/MBTI术语表.md`，检查所有非 active 页面中的专业术语翻译是否与术语表一致
- 重点关注：维度名称（心智/能量/本性/行动/身份）、特质名称（内倾/外倾/直觉/实感/思考/情感/计划/灵活/自信/动荡）、角色分组与策略分组名称
- 不一致的术语标注到该页面的完善建议中，并直接修正

## 四、结构完整性检查

### 4.1 index.md 同步

- 对比 `wiki/` 目录下的实际文件与 `wiki/wiki-index.md` 中的条目
- 实际存在但未在 wiki-index.md 中注册的页面 → 补充注册
- wiki-index.md 中列出但实际不存在的页面 → 标注为"待创建"或移除条目

### 4.2 交叉引用完整性

- 扫描所有 Wiki 页面中的 `[[双链]]`
- 检查链接目标页面是否存在
- 不存在的链接目标 → 标注为待创建或修正链接

### 4.3 孤立页面检测

- 找出没有任何其他页面通过双链指向的页面（wiki-index.md 和 log.md 除外）
- 孤立页面应被相关页面引用，或在 wiki-index.md 中确保可达

### 4.4 来源摘要页完整性

- 对照 `units/index.md` 中已有的来源文件夹，检查 `wiki/sources/` 是否有对应的摘要页
- 缺失的来源摘要页 → 标注为待创建

## 五、检查产出

每次维护检查完成后，输出以下内容：

### 检查报告格式

```
## 维护检查报告 — [日期]

### review 页面（X 个）
- [页面名]：[发现的问题] → [已修正/待人类确认]

### unreviewed 页面（X 个）
- [页面名]：建议 [保留/合并/补充/删除] — [理由]

### draft 页面（X 个）
- [页面名]：[完善建议]

### 结构问题
- [问题描述] → [修正操作]

### 建议用户操作
- 审核以下页面：[列表]
- 补充以下来源资料：[列表]（如有缺口）
```

### 日志记录

在 `wiki/log.md` 中追加维护检查记录：

```markdown
## [YYYY-MM-DD] lint | 维护检查
- 检查 review 页面 X 个，draft 页面 X 个，unreviewed 页面 X 个
- [主要发现和操作摘要]
```
