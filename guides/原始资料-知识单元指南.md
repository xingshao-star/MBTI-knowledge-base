# 原始资料 → 知识单元 拆解指南

> 本指南是收录原始资料的唯一执行指南。
> 适用于所有来源类型：书籍、文章、测试量表、案例等。

---

## 一、核心原则

1. **每个知识单元 = 一个独立 MD 文件**，存放于 `units/{来源文件夹}/` 对应子目录
2. **以语义完整性为拆分标准**，不以字数为标准
3. 每个单元聚焦一个细分领域，内容自洽可独立理解
4. 若单元内容中存在与其他知识单元的语义关联，应合理解耦，确保每个知识单元相对独立
5. 拆分后的子单元必须保留原章节的标签和分类上下文

---

## 二、收录前准备

### 查阅来源清单

收录任何资料前，先阅读 `raw/来源清单.md`：
- 确认该资料是否已登记
- 了解其来源类型、是否属于系列丛书、内容结构特征
- 如未登记，先请人类补充来源清单

### 来源未登记时的处理

当待收录资料尚未在 `raw/来源清单.md` 中登记，且文件不在 `raw/` 对应子目录时：

1. **迁移文件**：将原始资料移动到 `raw/` 下对应的子目录（书籍 → `raw/books/`、文章 → `raw/articles/`）
2. **登记来源清单**：在 `raw/来源清单.md` 对应类别表格中新增条目，填写文件名、系列丛书归属、内容结构、涉及类型，状态设为 `待收录`
3. 登记完成后，继续执行后续收录流程

### 创建来源文件夹

在 `units/` 下创建以来源文件名（去扩展名）命名的文件夹，作为该来源所有知识单元的存放根目录：

1. 文件夹名 = `raw/` 下对应文件的文件名去扩展名（如 `ENFJ-Protagonist主人公.pdf` → `ENFJ-Protagonist主人公/`）
2. 按需创建子目录：`{TYPE}/`（类型专属单元）、`relationships/`（关系单元）
3. 通用理论内容（维度、类型组、策略、理论历史等）直接放在来源文件夹根目录

### 建立来源双链

收录资料前，需在原始资料文件与 `raw/来源清单.md` 之间建立 Obsidian 双向链接：

1. **来源清单 → 原始资料**：在 `raw/来源清单.md` 的表格中，将"文件名"列的值改为 `[[文件名（去扩展名）]]` 格式
   - 示例：`ENFJ-Protagonist主人公.md` → `[[ENFJ-Protagonist主人公]]`
2. **原始资料 → 来源清单**：在原始资料文件开头插入反向链接
   - 格式：`> 来源登记：[[来源清单]]`

### 判断资料类型

| 来源类型 | 特征 | 拆分策略 |
|----------|------|----------|
| **系列丛书** | 多本书结构类似 | 每本独立拆解全部内容；关系内容跨书合并 |
| **独立书籍** | 单本完整著作，无系列关联 | 按标题+语义完整拆解，无需考虑跨书去重 |
| **文章** | 篇幅较短，主题聚焦 | 通常整篇为一个或少数几个知识单元 |
| **测试量表** | 结构化题目与评分标准 | 按测试维度拆分 |
| **案例** | 具体人物或情境分析 | 按分析维度拆分，标注涉及类型 |

## 三、拆分方法

### 第一步：通读与结构分析

通读资料，输出：
- 章节结构大纲
- 对于系列丛书：标记各章节的内容类型（通用理论 / 类型专属 / 关系内容）
- 对于独立资料：标记主要主题和细分领域

### 第二步：按标题层级初步切分

以资料的 `#`（一级标题）和 `##`（二级标题）作为主要切分边界：
- 每个 `##` 标题下的内容初步视为一个候选知识单元
- `#` 一级标题作为分类（category）标记
- 文章类资料若无明确标题层级，按段落语义切分

### 第三步：语义评估与细分

对每个候选单元进行语义评估：

- **单一领域** → 直接作为一个知识单元
- **包含多个细分领域** → 进一步拆分为多个子单元，每个子单元：
  - 聚焦一个细分主题
  - 继承父章节的 category 和 tags
  - id 中追加细分后缀

**语义拆分判断标准：**
- 内容是否讨论了不同的子主题（如"平衡状态"与"失衡状态"）
- 内容是否可以独立回答不同类型的问题
- 内容是否涉及不同的应用场景或对象

### 第四步：关系内容特殊处理

恋爱关系、友谊、职场协作等涉及两个类型互动的内容：
- 提取到 `units/{来源文件夹}/relationships/` 统一管理
- 文件命名：`{TYPE-A}-{TYPE-B}-{场景}.md`（按字母序排列类型代码）
- 当多份资料包含同一段关系的描述时，合并为一个单元，用 `## {TYPE} 视角` 分别保留各方论述
- 如果只处理了一方资料，先创建单视角单元，后续合并

## 四、Frontmatter 结构

> 创建知识单元时，以 `templates/knowledge-unit.md` 为骨架模板。

```yaml
---
id: INFJ-growth-self-esteem-balanced
type: INFJ
category: personal-growth
topic: 自尊 — 平衡状态
tags: [自尊, 平衡, 使命感, 和谐]
related_units:
  - INFJ-growth-self-esteem-imbalanced
  - shared-dimension-introversion
# ─── 溯源字段（仅供定位查询，不纳入用户回答内容） ───
_source: raw/books/INFJ-Advocate zh-Hans.txt
_source_lines: 455-480
_word_count: 850
---
```

### 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| `id` | 是 | 唯一标识，格式：`{type}-{category}-{topic简写}` |
| `type` | 是 | 所属人格类型代码，通用内容填 `shared` |
| `category` | 是 | 分类，见下方分类表 |
| `topic` | 是 | 具体主题名称 |
| `tags` | 是 | 检索标签，3-8 个关键词 |
| `related_units` | 否 | 关联知识单元的 id 列表 |
| `_source` | 是 | 来源文件路径（`_` 前缀表示非用户可见） |
| `_source_lines` | 是 | 来源行号范围 |
| `_word_count` | 是 | 单元字数 |

> **重要**：以 `_` 开头的溯源字段记录在结构中作为必要时的查询指引，但 LLM 在回答用户问题时不得将这些信息整理进入回答内容。

### 分类（category）取值

| category 值 | 适用内容 |
|-------------|----------|
| `theory` | 理论概述（五大维度、类型组、角色策略） |
| `overview` | 类型总览与核心特征 |
| `dimension-trait` | 维度在具体类型上的表现 |
| `personal-growth` | 个人成长（自尊、自信、自我进化等） |
| `romantic` | 恋爱关系 |
| `friendship` | 友谊 |
| `parenting` | 育儿 |
| `academic` | 学术路径 |
| `career` | 职业发展与职场协作 |

> 当资料内容不属于以上任何分类时，可新增 category 值，并在 `units/知识单元index.md` 中注册。

## 五、处理流程

### 通用流程（所有来源类型）

```
1. 查阅 raw/来源清单.md → 了解资料特征
   · 如未登记：迁移文件至 raw/ 对应子目录，并在来源清单中新增条目
   · 建立来源双链：来源清单表格中文件名改为 [[文件名]] 格式，原始资料文件开头插入 [[来源清单]] 反向链接
2. 创建来源文件夹 → units/{来源文件名去扩展名}/，按需创建子目录
3. 通读资料 → 输出结构大纲
4. 按标题 + 语义拆分为知识单元
5. 为每个单元填写完整 Frontmatter
6. 对单元内容依据内容进行润色以便更准确的以中文来描述表达
7. 术语校对 → 参照 `docs/MBTI术语表.md`，逐一核对单元中的专业术语翻译是否与术语表一致；不一致的统一替换为术语表中的标准译法
8. 创建来源知识单元清单 → 在来源文件夹下创建 `{TYPE} 知识单元清单.md`，列出该来源产出的所有知识单元，并添加指向全局索引的反向链接 `[[知识单元index|知识单元索引]]`
9. 更新 units/知识单元index.md → 在全局索引中注册该来源文件夹，确保与清单文件形成双向链接
10. 校验 → 无遗漏、元数据一致、来源行号准确、术语一致
```

### 系列丛书附加步骤

```
· 每本书独立拆解全部内容（包括通用理论），存入各自的来源文件夹
· 相似的通用理论内容在 WIKI 编译阶段通过综合编译/共识机制处理
· 关系内容：检查已有单视角单元，有则合并为双视角
```

## 六、命名规范

### 文件命名

- 维度单元：`units/{来源文件夹}/{维度前缀}维度-{中文特质1}（{代码1}）与{中文特质2}（{代码2}）.md`
  - 示例：`units/ENFJ-Protagonist主人公/心智维度-内倾型（I）与外倾型（E）.md`
  - 示例：`units/ENFJ-Protagonist主人公/身份维度-自信型（-A）与动荡型（-T）.md`
- 策略单元：`units/{来源文件夹}/策略-{标准中文译名}（{维度1}×{维度2}）.md`
  - 策略由心智维度（I/E）与身份维度（-A/-T）交叉形成
  - 示例：`units/ENFJ-Protagonist主人公/策略-自信独行型（内倾型×自信型）.md`
  - 示例：`units/ENFJ-Protagonist主人公/策略-人际掌控型（外倾型×自信型）.md`
- 类型组单元：`units/{来源文件夹}/类型组-{代码}{中文名}.md`
  - 示例：`units/ENFJ-Protagonist主人公/类型组-NT分析家.md`
- 其他通用单元：`units/{来源文件夹}/{topic简写}.md`
  - 示例：`units/ENFJ-Protagonist主人公/MBTI理论-历史背景.md`
- 类型专属单元：`units/{来源文件夹}/{TYPE}/{category}-{topic简写}.md`
  - 示例：`units/ENFJ-Protagonist主人公/ENFJ/成长-自尊.md`
- 关系单元：`units/{来源文件夹}/relationships/{TYPE-A}-{TYPE-B}-{场景}.md`
  - 示例：`units/ENFJ-Protagonist主人公/relationships/ENFJ-ISTP-恋爱.md`

### id 命名

- 格式：`{type}-{category}-{topic简写}`
- 使用大写类型代码 + 小写英文连字符
- 示例：`INFJ-growth-self-esteem-balanced`、`shared-theory-dimension-EI`

## 七、来源知识单元清单

每个来源文件夹下必须包含一个知识单元清单文件（命名格式：`{TYPE} 知识单元清单.md`），列出该来源拆解产生的所有知识单元。

### 创建时机

在来源文件夹内所有知识单元创建完毕后，生成此清单文件，作为该来源的目录索引。

### 格式规范

详细模板参见 `templates/知识单元清单模板.md`，以下是核心约定：

#### 文件头

```markdown
# {来源文件夹名} — 知识单元清单

> 来源：`raw/{子目录}/{文件名}`
> 本文件列出该来源拆解产生的所有知识单元。
> 返回：[[知识单元index|知识单元索引]]
```

- 来源行中 `{子目录}` 为 `raw/` 下的实际存放位置（`books`、`articles` 等）
- 返回链接统一指向 `[[知识单元index|知识单元索引]]`

#### 条目格式

```
- `{unit-id}` — [{文件名}]({相对路径})
```

- `{unit-id}` 与该单元的 frontmatter `id` 字段完全一致
- `{文件名}` 仅写文件名（不含路径），`{相对路径}` 为相对于本清单文件的路径
- 共享内容引用路径固定为 `../人格共享内容/{文件名}`

#### 章节组织

| 章节 | 何时出现 | 内容 |
|------|----------|------|
| `## 理论背景` | 来源含 MBTI 理论历史等通用背景 | 共享内容引用 |
| `## 五大维度` | 来源涉及五个维度的描述 | 5条 `shared-dimension-*` 引用 |
| `## 四大角色组` | 来源涉及 NT/NF/SJ/SP 角色组 | `shared-theory-roles` + 4条 `shared-typegroup-*` |
| `## 四大策略` | 来源涉及四大策略描述 | `shared-theory-strategies` + 4条 `shared-strategy-*` |
| `## {TYPE} 类型专属` | 必含 | 该类型所有专属知识单元 |
| `## 关系单元` | 存在 relationships/ 子目录时 | 按场景分组的跨类型关系单元 |

**灵活性原则：**

- **通用理论章节根据实际内容增减**——不同书籍/来源包含的理论内容不同。如来源不含策略内容，则省略 `## 四大策略` 章节；如含其他通用内容（如"类型与特质"），则在理论背景中追加对应条目
- **共享内容的引用文件名以实际文件为准**——每个来源的共享内容文件名可能不同（部分存在个性化命名），清单中必须使用实际存在的文件名
- **类型专属的 category 小节按资料实际结构组织**——不同书籍的章节划分方式不同，category 列表可随资料内容扩展（如出现 `## 五大维度` 之外的维度章节、`conclusion` 总结章节等），不强制限定于固定 category 集合
- **关系单元的场景分组按实际文件组织**——常见为恋爱 / 友谊 / 职场协作三层，若来源含"育儿"等额外场景则追加对应分组

#### 排序规则

- 通用理论章节按"理论背景 → 五大维度 → 四大角色组 → 四大策略"顺序
- 类型专属内 category 小节按字母序排列（`academic → career → friendship → overview → parenting → personal-growth → romantic`）
- 如有非标准 category，追加在标准 category 之后
- 关系单元内场景分组按"恋爱 → 友谊 → 职场协作 → （其他）"顺序

#### ID 命名约定

| 前缀 | 格式 | 示例 |
|------|------|------|
| 共享维度 | `shared-dimension-{ie\|ns\|tf\|jp\|at}` | `shared-dimension-ie` |
| 共享角色组 | `shared-typegroup-{nt\|nf\|sj\|sp}` | `shared-typegroup-nt` |
| 共享策略 | `shared-strategy-{简写}` | `shared-strategy-confident-individualism` |
| 共享其他理论 | `shared-theory-{简写}` | `shared-theory-history` |
| 类型专属 | `{TYPE}-{category}-{topic}` | `INFJ-growth-self-esteem` |
| 关系单元 | `{TYPE}-{NT\|NF\|SJ\|SP}-{场景}` | `INFJ-NT-romantic` |

### 完整性要求

1. 清单中每条必须对应一个实际存在的 `.md` 文件
2. 来源文件夹内每个 `.md` 文件（除清单自身外）均须在清单中有对应条目
3. 必须包含指向全局索引的反向链接，确保 Obsidian 知识图谱中形成双向链接
4. 清单文件的命名格式固定为 `{TYPE} 知识单元清单.md`，其中 `{TYPE}` 为该来源的主人格类型代码

### 校验清单

- [ ] 共享内容引用文件名与实际文件一致（特别注意中文括号、特殊字符）
- [ ] 类型专属条目数 = 来源 `{TYPE}/` 子目录下文件数
- [ ] 关系单元条目数 = 来源 `relationships/` 子目录下文件数
- [ ] 所有 `unit-id` 与对应 frontmatter `id` 一致
- [ ] 返回链接 `[[知识单元index|知识单元索引]]` 存在
- [ ] 无重复条目、无空链接
