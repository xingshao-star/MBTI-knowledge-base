# LIMINA 知识库 — 知识库架构说明

> 本文档说明 MBTI 知识库的架构设计、目录结构与运作流程。
> 本文档供人类了解知识库全貌，LLM 回答用户问题时不需要查阅本文档。

---

## 一、设计理念

传统 RAG 方式是"每次提问都从原始文档重新检索拼凑答案"，知识不会积累。

LLM Wiki 模式的核心区别：**LLM 渐进式地构建和维护一个持久化的 Wiki**。每收录一份新素材，LLM 不只是索引它，而是阅读、提取、整合到已有的知识体系中。

**改进点**：针对系列化长篇书籍，在原始素材与 Wiki 之间增加"知识单元层"，解决：
1. 单本书籍体量过大，直接编译 Wiki 效果差
2. 系列书籍含大量重复通用内容，需去重
3. 查询补充时需要比整本书更精准的检索粒度

三个角色分工：
- **人类**：策展素材、提出问题、引导方向、审核确认
- **LLM**：拆解资料、编译 Wiki、回答查询、维护一致性
- **Obsidian**：可视化浏览 Wiki，查看图谱视图，跟踪页面关联

## 二、四层架构

```
MBTI知识库/
├── CLAUDE.md        ← 全局入口（任务路由）
├── raw/             ← 第一层：原始素材（人类管理，LLM 只读）
├── units/           ← 第二层：知识单元（LLM 生成，人类可审阅）
├── wiki/            ← 第三层：知识 Wiki（LLM 维护）
├── templates/       ← 辅助层：页面模板（由指南引用）
├── docs/            ← 辅助层：架构说明文档（人类参考）
└── guides/          ← 辅助层：LLM 执行指南
```

### 第一层：raw/（原始素材）

**所有者**：人类
**规则**：不可变，LLM 只读不写。查询阶段不可访问，仅在拆解阶段读取。

| 子目录 | 存放内容 |
|--------|----------|
| `raw/books/` | 书籍、论文、专著 |
| `raw/articles/` | 网络文章、博客、专栏 |
| `raw/来源清单.md` | 所有资料的登记清单（系列标记、结构特征） |

### 第二层：units/（知识单元）

**所有者**：LLM 生成，人类可审阅
**规则**：每个知识单元为一个独立 MD 文件，带结构化 Frontmatter。按来源文件夹组织，每份原始资料的拆解产物存放在以来源文件名（去扩展名）命名的文件夹中。

| 子目录 | 内容 |
|--------|------|
| `units/index.md` | 全局索引（指向各来源文件夹） |
| `units/{来源文件夹}/` | 每份来源一个文件夹 |
| `units/{来源文件夹}/index.md` | 该来源的知识单元清单 |
| `units/{来源文件夹}/{TYPE}/` | 各类型专属单元 |
| `units/{来源文件夹}/relationships/` | 关系单元 |
| `units/{来源文件夹}/*.md` | 通用理论单元（维度、类型组、策略等） |

### 第三层：wiki/（知识 Wiki）

**所有者**：LLM
**规则**：LLM 全权负责创建、更新、维护；人类只读浏览

| 目录 | 内容 |
|------|------|
| `wiki/overview.md` | MBTI 体系叙事性综述（800-1500 字） |
| `wiki/types/{TYPE}/` | 每个类型一个子目录：枢纽页（总览+导航）+ 详情页（1:1 对应知识单元） |
| `wiki/dimensions/` | 四大维度 + A/T 身份维度 |
| `wiki/functions/` | 八大认知功能 |
| `wiki/type-groups/` | 四大类型组：NF型人、NT型人、SJ型人、SP型人 |
| `wiki/strategies/` | 四大策略（心智维度×身份维度交叉） |
| `wiki/theory/` | 理论背景（历史、方法论） |
| `wiki/relationships/` | 类型间关系（组级别 + 个体级别） |
| `wiki/growth/` | 跨类型个人成长专题 |
| `wiki/life-domains/` | 跨类型生活领域应用 |
| `wiki/comparisons/` | 对比分析 |
| `wiki/sources/` | 来源摘要 |
| `wiki/wiki-index.md` | 语义目录（每条含摘要，LLM 查阅入口） |
| `wiki/log.md` | 操作日志 |

### 第四层：CLAUDE.md（Schema）

LLM 的操作手册，包含目录结构、页面约定、工作流程引用、内容规范。

### 辅助层

| 目录 | 用途 |
|------|------|
| `templates/` | 页面模板（由指南明确引用，创建页面时作为骨架） |
| `guides/` | LLM 执行指南（三份，各司其职） |
| `docs/` | 架构说明文档（人类参考，LLM 不主动读取） |

## 三、指南体系

三份指南各有明确职责，互不重叠：

| 指南 | 路径 | 职责 | 使用时机 |
|------|------|------|----------|
| 原始资料→知识单元 | `guides/原始资料-知识单元指南.md` | 收录资料的唯一指南 | 拆解原始资料时 |
| 知识单元→Wiki | `guides/知识单元-WIKI指南.md` | 发布/编译 Wiki 的唯一指南 | 从知识单元生成 Wiki 时 |
| LLM 查阅知识库 | `guides/LLM查阅知识库指南.md` | 回答用户问题的唯一查阅指南 | 用户提问时 |

## 四、运作流程

### 收录流程（两阶段）

```
阶段一：原始资料 → 知识单元
  1. 查阅 raw/来源清单.md
  2. 阅读 guides/原始资料-知识单元指南.md
  3. 创建来源文件夹 units/{来源文件名去扩展名}/
  4. 通读资料 → 拆解为知识单元 → 存入来源文件夹
  5. 创建来源 index.md，更新全局 units/index.md

阶段二：知识单元 → Wiki（两种模式）
  1. 阅读 guides/知识单元-WIKI指南.md
  2. 指定来源文件夹，读取来源 index.md 获取编译清单
  3. 单来源 → 轻量发布（替换 frontmatter + 添加双链）
     多来源 → 综合编译（合并 + 共识机制）
  4. 类型页面：枢纽页（总览+导航）+ 详情页（1:1 对应单元）
  5. 更新 wiki/wiki-index.md（语义目录）和 wiki/log.md
```

### 查询流程

```
用户提问
  → LLM 阅读 guides/LLM查阅知识库指南.md
  → 两级检索：wiki → units（raw 不可访问）
  → 渐进式双链查阅（精确/探索/开放三种模式）
  → Wiki 无匹配时：从知识单元补充，同步创建 unreviewed 条目
```

### 维护流程

定期检查：交叉引用完整性、孤立页面、矛盾检测、索引同步、未审核页面

### 审核流程

```
LLM 查询中创建的页面 → status: unreviewed
  → 人类在 Obsidian 中审阅
  → 确认准确 → 改为 active
  → 需修正 → 告知 LLM 修改后改为 active
  → 不需要 → 告知 LLM 删除
```

## 五、页面状态定义

| 状态 | 含义 | 判定条件 | 谁可设置 |
|------|------|----------|----------|
| `draft` | 收录编译中创建 | LLM 执行收录流程时系统性创建 | LLM |
| `active` | 已审核确认 | 人类审阅通过 | 仅人类 |
| `unreviewed` | 查询动态创建 | LLM 回答用户问题时自动创建 | LLM |
| `review` | 需重新审核 | 已有 active 页面因新来源导致实质性变更 | LLM |
